Network Troubleshooting Overview

Network troubleshooting in AI infrastructure requires a systematic methodology to identify, isolate, and resolve connectivity and performance issues. Effective troubleshooting minimizes downtime, ensures reliable data transfer for training workloads, and maintains optimal network performance. A structured approach following the OSI model layers and using appropriate diagnostic tools enables efficient problem resolution in complex AI network environments.

Troubleshooting Methodology

A systematic troubleshooting methodology ensures efficient problem resolution. Follow a structured approach from problem identification to solution verification.

1Information Gathering

Collect comprehensive information about the network issue:

  • Symptoms: What is the observed problem? (No connectivity, slow performance, packet loss)
  • Scope: Is the issue affecting one node, multiple nodes, or the entire cluster?
  • Timeline: When did the issue start? Was it sudden or gradual?
  • Context: What changed recently? (Hardware changes, configuration changes, software updates)
  • Error Messages: Collect all error messages and log entries
  • Affected Services: Which services or applications are impacted?

2Problem Isolation

Isolate the problem to identify the root cause:

  • Single Node vs Multiple Nodes: Determine if issue is node-specific or cluster-wide
  • Network Segment: Identify which network segment is affected (management, data, storage)
  • OSI Layer: Determine which OSI layer is experiencing issues
  • Hardware vs Software: Determine if issue is hardware or software related
  • Reproducibility: Determine if issue is reproducible or intermittent

3OSI Layer Analysis

Analyze the problem using the OSI model layers:

  • Layer 1 (Physical): Cables, connectors, ports, link status
  • Layer 2 (Data Link): MAC addresses, VLANs, switches, link aggregation
  • Layer 3 (Network): IP addresses, routing, subnets, gateways
  • Layer 4 (Transport): TCP/UDP, ports, connections, flow control
  • Layer 5-7 (Upper Layers): Applications, protocols, services

4Hypothesis Formation and Testing

Form and test hypotheses systematically:

  • List possible causes ranked by likelihood
  • Test one hypothesis at a time
  • Use diagnostic tools to gather evidence
  • Document test results
  • Start with simplest solutions first

5Solution Implementation and Verification

Implement solution and verify resolution:

  • Plan the solution approach
  • Implement solution in controlled manner
  • Verify solution resolves the issue
  • Monitor for recurrence
  • Document the solution for future reference

OSI Model-Based Troubleshooting

Using the OSI model provides a structured approach to network troubleshooting. Each layer has specific checks and diagnostic procedures.

Layer 1: Physical Layer

Checks:

  • Verify cable connections (InfiniBand, Ethernet)
  • Check link status (link up/down)
  • Inspect cables for damage
  • Verify port status on switches
  • Check transceiver/module status
  • Verify power to network equipment
# Check link status (InfiniBand) ibstat iblinkinfo # Check link status (Ethernet) ip link show ethtool [interface] # Check port status # InfiniBand: ibswitches, ibhosts # Ethernet: switch management interface

Layer 2: Data Link Layer

Checks:

  • Verify MAC address assignment
  • Check VLAN configuration
  • Verify switch port configuration
  • Check link aggregation (LACP) status
  • Verify InfiniBand LID assignment
  • Check for MAC address conflicts
# Check MAC addresses ip link show ifconfig # Check VLAN configuration ip -d link show bridge vlan show # Check InfiniBand LID assignment ibstat ibhosts # Check link aggregation ip link show bond0 cat /proc/net/bonding/bond0

Layer 3: Network Layer

Checks:

  • Verify IP address assignment
  • Check routing tables
  • Test connectivity (ping)
  • Verify subnet configuration
  • Check default gateway
  • Verify InfiniBand GID assignment
# Check IP configuration ip addr show ip route show # Test connectivity ping [target_ip] ping6 [target_ipv6] # Check routing ip route get [destination] traceroute [destination] # Check InfiniBand GID ibstat -g

Layer 4: Transport Layer

Checks:

  • Verify TCP/UDP connectivity
  • Check port availability
  • Test RDMA connectivity (InfiniBand)
  • Check firewall rules
  • Verify connection state
# Test TCP connectivity telnet [host] [port] nc -zv [host] [port] # Test RDMA connectivity (InfiniBand) ibping [target_lid] ib_write_bw [target] # Check listening ports netstat -tuln ss -tuln # Check firewall iptables -L firewall-cmd --list-all

InfiniBand Troubleshooting

InfiniBand troubleshooting requires understanding InfiniBand-specific components and diagnostic tools. Common issues include link problems, subnet manager issues, routing problems, and performance degradation.

Link Status Issues

Symptoms: No connectivity, link down, degraded performance

Troubleshooting Steps:

  1. Check physical connections (cables, connectors)
  2. Verify link status: ibstat
  3. Check link width and speed: ibstat -v
  4. Verify port state: iblinkinfo
  5. Check for link errors: ibstat -s
  6. Test with different cables/ports
  7. Check switch port status
# Check InfiniBand link status ibstat ibstat -v # Check link information iblinkinfo # Check port state ibstat -s # Check for errors ibstat -e

Subnet Manager Issues

Symptoms: Devices not discovered, LID assignment failures, routing problems

Troubleshooting Steps:

  1. Verify subnet manager is running: systemctl status opensm
  2. Check subnet manager logs
  3. Verify LID assignment: ibstat
  4. Check GID assignment: ibstat -g
  5. Restart subnet manager if needed
  6. Check for multiple subnet managers (conflict)
# Check subnet manager status systemctl status opensm opensm -D # Check LID assignment ibstat | grep -i lid # Check GID assignment ibstat -g # View subnet manager logs tail -f /var/log/opensm.log

Performance Issues

Symptoms: Low bandwidth, high latency, degraded performance

Troubleshooting Steps:

  1. Measure bandwidth: ib_write_bw
  2. Measure latency: ib_write_lat
  3. Check link speed and width
  4. Verify routing paths: ibtracert
  5. Check for congestion
  6. Verify SHARP configuration (if used)
  7. Check UFM for fabric-wide issues
# Measure bandwidth ib_write_bw [target] # Measure latency ib_write_lat [target] # Trace route ibtracert [target_lid] # Check fabric topology ibnetdiscover # Use UFM for comprehensive analysis # (UFM provides detailed performance metrics)

Ethernet Troubleshooting

Ethernet troubleshooting in AI infrastructure focuses on connectivity, performance, configuration, and QoS issues. Common problems include link failures, performance degradation, and configuration errors.

Connectivity Issues

Symptoms: No connectivity, intermittent connectivity, link down

Troubleshooting Steps:

  1. Check link status: ip link show
  2. Verify interface is up: ip link set [interface] up
  3. Check cable connections
  4. Test connectivity: ping [target]
  5. Verify IP configuration: ip addr show
  6. Check routing: ip route show
  7. Verify switch port status
# Check link status ip link show ethtool [interface] # Bring interface up ip link set [interface] up # Test connectivity ping -c 4 [target_ip] # Check IP configuration ip addr show [interface] # Check routing ip route show

Performance Issues

Symptoms: Low throughput, high latency, packet loss

Troubleshooting Steps:

  1. Measure bandwidth: iperf3
  2. Check for packet loss: ping -f
  3. Verify link speed: ethtool [interface]
  4. Check for errors: ethtool -S [interface]
  5. Monitor interface statistics
  6. Check for congestion
  7. Verify QoS configuration
# Measure bandwidth iperf3 -c [server] -t 60 # Check for packet loss ping -f -c 1000 [target] # Check link speed and errors ethtool [interface] ethtool -S [interface] # Monitor interface statistics watch -n 1 'cat /proc/net/dev'

Configuration Issues

Symptoms: Incorrect IP, routing problems, VLAN issues

Troubleshooting Steps:

  1. Verify IP configuration
  2. Check subnet mask
  3. Verify default gateway
  4. Check VLAN configuration
  5. Verify DNS configuration
  6. Check network configuration files
# Check IP configuration ip addr show ip route show # Check DNS cat /etc/resolv.conf nslookup [hostname] # Check network configuration # NetworkManager: nmcli connection show # systemd-networkd: networkctl status

Diagnostic Tools

Effective network troubleshooting requires appropriate diagnostic tools. Use tools specific to InfiniBand and Ethernet networks for comprehensive diagnostics.

InfiniBand Diagnostic Tools

  • ibstat: Display InfiniBand device status
  • ibping: Test InfiniBand connectivity
  • iblinkinfo: Display link information
  • ibnetdiscover: Discover fabric topology
  • ibtracert: Trace InfiniBand routes
  • ib_write_bw: Measure bandwidth
  • ib_write_lat: Measure latency
  • ibhosts/ibswitches: List hosts/switches
  • UFM: Unified Fabric Manager for comprehensive monitoring

Ethernet Diagnostic Tools

  • ip/ifconfig: Interface configuration and status
  • ethtool: Ethernet device settings and statistics
  • ping: Test connectivity
  • traceroute: Trace network path
  • iperf3: Measure bandwidth
  • netstat/ss: Network connections and statistics
  • tcpdump: Packet capture and analysis
  • wireshark: Protocol analysis

Performance Monitoring Tools

  • iftop: Real-time bandwidth monitoring
  • nload: Network traffic monitor
  • vnstat: Network statistics
  • UFM: InfiniBand fabric monitoring
  • Prometheus/Grafana: Network metrics collection and visualization

Common Network Issues and Solutions

Understanding common network issues and their solutions helps resolve problems quickly. Many issues follow predictable patterns.

Issue: No Connectivity

Possible Causes:

  • Physical layer problems (cables, connectors)
  • Interface down
  • Incorrect IP configuration
  • Routing problems
  • Firewall blocking

Solutions:

  • Check physical connections
  • Bring interface up: ip link set [interface] up
  • Verify IP configuration
  • Check routing tables
  • Verify firewall rules

Issue: Slow Performance

Possible Causes:

  • Link speed limited
  • Network congestion
  • Packet loss
  • Incorrect routing
  • QoS misconfiguration

Solutions:

  • Verify link speed: ethtool [interface]
  • Check for congestion
  • Investigate packet loss
  • Optimize routing
  • Review QoS configuration

Issue: Intermittent Connectivity

Possible Causes:

  • Loose cable connections
  • Faulty cables or connectors
  • Switch port issues
  • Link aggregation problems
  • Driver issues

Solutions:

  • Reseat cables
  • Replace faulty cables
  • Check switch port status
  • Verify link aggregation configuration
  • Update drivers
Troubleshooting Best Practices: 1. Document everything (symptoms, tests, results)
2. Test one variable at a time
3. Start with physical layer (Layer 1)
4. Use systematic OSI model approach
5. Use appropriate diagnostic tools
6. Consult documentation and knowledge bases
7. Escalate when stuck
8. Learn from resolved issues
9. Maintain network documentation
10. Implement monitoring to prevent issues
